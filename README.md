# Scan2BIM Converter

Desktop app that converts E57 point clouds to Potree v2 octrees for the
Scan2BIM web viewer. Same pipeline as the Part 1 Docker image (PDAL +
PotreeConverter), running locally on the user's machine.

Single repo, three target platforms, all builds produced by GitHub Actions.

## Layout

```
scan2bim-converter/
  src-tauri/                # Rust backend (Tauri 2 app shell)
    src/
      main.rs               # entry point, defers to lib::run()
      lib.rs                # Tauri command handlers (pick file, pick output, ...)
      converter.rs          # the actual two-step conversion pipeline
    Cargo.toml              # Rust deps + the 'ubuntu20' feature flag
    tauri.conf.json         # window config, identifier, icon refs
    tauri.bundle.conf.json  # overlay applied at 'tauri build' time
    capabilities/default.json   # Tauri 2 permission scope
    icons/                  # generated by `npx tauri icon`
  renderer/                 # Tauri webview UI (vanilla HTML/CSS/JS)
    index.html              # all 6 UI states in one DOM, toggled via [hidden]
    styles.css              # dark+amber glassmorphism theme, Manrope font
    app.js                  # state machine + i18n + IPC calls
  scripts/
    package-windows.ps1     # post-build: assembles the portable Windows ZIP
    package-linux.sh        # post-build: assembles the portable Linux tar.gz
    package-macos.sh        # post-build: assembles the portable macOS zip
  .github/workflows/
    build-potreeconverter.yml   # CI for all platforms (PotreeConverter + app)
  package.json              # npm scripts for dev/build/package
```

## How conversion works

`renderer/app.js` listens for the Convert button, calls
`invoke('convert_start', ...)` via the Tauri JS bridge.

`src-tauri/src/converter.rs::run_pipeline` is the entry point:

1. **If input is .e57**: spawn `pdal translate input.e57 /tmp/scan2bim-<id>.las`.
   Drives the 0-48% portion of the progress bar by polling the LAS file size
   against a ~2.5x size estimate.
2. **Then always**: spawn `PotreeConverter -i <las> -o <out> --encoding BROTLI`.
   Parses stdout for `[NN%, Ns]` lines, maps to 50-100% of the progress bar
   (or 0-100% if no PDAL step was needed). Streams stdout + stderr through
   `convert:log` events so the Logs modal in the renderer can show them.
3. **On success**: emits `convert:done`. On failure: `convert:error` with a
   code (`pdal_exit_N`, `potree_exit_N`, `empty_output`, etc.) the renderer
   shows on the Error screen.

Bundled binaries live in `scan2bim-converter/binaries/`:

```
binaries/
  PotreeConverter[.exe]     # the LAS-to-Potree binary, per platform
  laszip.dll | liblaszip.so.* | liblaszip.dylib    # sibling runtime dep
  libtbb.so.12              # Linux only; needed by PotreeConverter on Linux
  pdal/                     # PDAL bundle (E57-to-LAS step)
    bin/pdal[.exe]
    lib/                    # gdal, geos, proj, libe57format, ... all the C++ deps
    share/gdal              # GDAL CRS data
    share/proj              # PROJ grid files
```

`converter.rs::locate_binary` finds these at dev time relative to
`CARGO_MANIFEST_DIR` and at runtime relative to `current_exe()`. When
spawning the children, `apply_pdal_env` / `apply_potree_env` set the
right `PATH` / `GDAL_DATA` / `PROJ_LIB` / `LD_LIBRARY_PATH` /
`DYLD_LIBRARY_PATH` so the bundle is self-contained.

## Running locally (dev mode, watches files)

Windows / Linux / macOS, from the repo root:

```
npm install
npm run dev
```

`npm run dev` calls `tauri dev`. On Windows requires VS Build Tools + Rust;
on Linux requires `libwebkit2gtk-4.1-dev` (Ubuntu 22.04+) or 4.0 (Ubuntu
20.04 via the `ubuntu20` feature, see below); on macOS just Xcode CLT.

## Building distributable archives

### Windows (host: any Windows with VS Build Tools + Rust + Node)

```
npm run build:windows
```

Produces `dist/Scan2BIM-Converter-<ver>-Windows-x64.zip`. Portable, no
installer. Drop on a fresh machine and run.

### Linux 22.04+ (host: Ubuntu 22.04+ or our CI)

```
npm run build:linux
```

Same idea, produces a `.tar.gz` in `dist/`. Requires
`libwebkit2gtk-4.1-dev` at build time and `libwebkit2gtk-4.1-0` at run
time on the target.

### Linux 20.04 (host: Ubuntu 20.04 or container)

```
cd src-tauri
cargo build --release --features ubuntu20
cd ..
bash scripts/package-linux.sh
```

The `ubuntu20` feature flips wry (Tauri's webview crate) from
`webkit2gtk-4-1` to `webkit2gtk-4-0` so the binary links against the
older WebKitGTK series Ubuntu 20.04 ships. CI runs this inside an
`ubuntu:20.04` Docker container so glibc is also pinned to 2.31.

### macOS ARM (host: Apple Silicon Mac)

```
npm run build:macos
```

Unsigned. First launch on a fresh Mac will be Gatekeeper-blocked;
right-click in Finder → Open → confirm dialog. Once is enough.

## CI

`.github/workflows/build-potreeconverter.yml` builds everything on a
push to anything under `.github/workflows/`, `scripts/`, `src-tauri/`,
`renderer/`, or `package.json`. Or trigger manually via "Run workflow"
in the Actions tab.

Jobs (each produces a downloadable artifact):

- `build-linux-x64` → `potreeconverter-linux-x64` (just the binary)
- `build-macos-arm` → `potreeconverter-darwin-arm64` (just the binary)
- `build-app-linux` (needs build-linux-x64) → `scan2bim-converter-linux-x64`
  (full portable tar.gz, requires WebKitGTK 4.1 on the user's machine)
- `build-app-linux-ubuntu20` (self-contained, no needs:) → `scan2bim-converter-linux-x64-ubuntu20`
  (full portable tar.gz, runs on Ubuntu 20.04 with WebKitGTK 4.0)
- `build-app-macos` (needs build-macos-arm) → `scan2bim-converter-macos-arm64`
  (full portable zip, unsigned)

Windows is currently built locally on Sohail's Windows machine — there's
no Windows job in CI because Tauri + MSVC + the PDAL bundle takes
significant setup on the GitHub Windows runners. To produce a fresh
Windows ZIP, run `npm run build:windows` from a Windows host with the
prerequisites installed.

## Common modifications

### Add a UI string in another language

`renderer/app.js` has an `i18n` dictionary at the top with `de` and
`en` keys. Add another top-level key for the new language. The DE/EN
toggle in `renderer/index.html` is hard-coded; add a third button there
too. Each translatable text in `index.html` carries a `data-i18n="key"`
attribute. Add new translations to all language dictionaries when
adding new UI strings.

### Change the conversion pipeline (e.g. add a PotreeConverter flag)

`src-tauri/src/converter.rs::run_potree_converter` builds the
`Command` for PotreeConverter. Add or change args there. Same pattern
for `run_pdal_translate` for the PDAL stage.

### Pass extra metadata into the produced Potree octree

The simplest path is post-processing: after PotreeConverter writes
`metadata.json`, add a step in `run_pipeline` (just before emitting
`convert:done`) that reads the JSON, adds your fields, writes it
back.

### Customize the window (size, frame, etc.)

`src-tauri/tauri.conf.json` → `app.windows[0]`. The mockup-driven
540x540 frameless is set there. Resizing or showing the title bar is
one flag flip.

### Change the bundled PDAL or PotreeConverter version

PDAL: conda install line in the CI workflow controls it. Change
`pdal` to e.g. `pdal=2.6.*` for a pinned version.

PotreeConverter: CI clones a fixed tag from GitHub
(`git clone --depth 1 --branch 2.1.2 ...`). Bump the branch arg to
the desired tag.

## Persistence

The renderer saves preferences (chosen output folder + DE/EN
language) to localStorage under the key `scan2bim:prefs:v1`. Cleared
by the user via browser-tools-style devtools, or by switching to a
fresh user profile. There is no other persistent state — each
conversion writes a fresh Potree folder.

## Where to look first when something goes wrong

- Conversion fails immediately: check the `Logs` button on the Error
  screen, then check the terminal that launched the app (release
  builds spawn no terminal, but `npm run dev` shows everything).
- Window doesn't open at all on Linux: missing `libwebkit2gtk-4.x-0`.
- macOS Gatekeeper blocks: right-click → Open the first time, or
  `xattr -dr com.apple.quarantine <folder>` from terminal.
- Windows: a child terminal pops up next to the conversion screen:
  the `CREATE_NO_WINDOW` flag in `converter.rs` got reverted; that
  flag is what keeps PDAL / PotreeConverter from spawning their own
  consoles.
